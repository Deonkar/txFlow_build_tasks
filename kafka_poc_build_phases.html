1
Get Kafka running locally — nothing else
Docker Compose with Redpanda (Kafka-compatible, single container, no Zookeeper headache). Open the web console, create the payments.initiated topic with 3 partitions, send a test message from the console, read it back. That's it.
2–3 days Docker Compose Redpanda Redpanda Console (UI)
What you'll actually understand after this: what a topic is, what a partition is, why messages are ordered within a partition but not across them, what an offset is (the message number), and why Kafka retains messages even after they're read.
2
FastAPI producer + Outbox pattern
POST /payment endpoint. Validates payload. Before publishing to Kafka, writes the event to an outbox table in PostgreSQL first — then publishes. If the app crashes after the DB write but before Kafka publish, the event survives. This is the Outbox pattern and it's load-bearing.
3–4 days FastAPI confluent-kafka-python PostgreSQL outbox table
What you'll understand: why you can't write to DB and Kafka atomically (dual-write problem), how the Outbox pattern solves it, what "at-least-once delivery" means and why that guarantee needs idempotency on the consumer side.
3
First consumer — wallet updater + Redis dedup
Write one Python consumer that reads payment events and updates a wallet balance in PostgreSQL. Before processing, check Redis: has this event_id been processed before? If yes, skip. If no, process and mark in Redis. Manual offset commit — only commit after successful processing, not before.
3–4 days aiokafka or confluent-kafka Redis dedup manual offset commit
What you'll understand: consumer groups, offset management (the hardest concept in Kafka), idempotency as a design constraint (not just a safety net), and why committing offsets before vs after processing changes the at-least-once guarantee.
4
Add the other 4 consumers
Fraud scorer (if amount > threshold, flag it). Notifier (print "email sent for user X"). Audit logger (write immutable record — never update, never delete, just append). Analytics (increment payment counters in Redis). Each is its own consumer group, each reads independently.
3–5 days 4 independent Python services 4 consumer groups
What you'll understand: this is the core Kafka lesson — one event, 5 independent readers, zero coupling. Kill the notifier service for an hour, restart it, and it replays from where it left off. That's not possible with SQS. That's Kafka's fundamental shift.
5
Retries + Dead Letter Queue
Add retry logic to each consumer (3 attempts with exponential backoff). If all 3 fail, publish the event to payments.dlq with error metadata attached. Write a DLQ consumer that logs failed events and shows why they failed.
2–3 days exponential backoff payments.dlq topic error envelope pattern
What you'll understand: why you can't just let consumers throw exceptions (it blocks the partition), how DLQ isolates bad events from good ones, and what "poison pill" events are — messages that will never succeed and need to be quarantined.
6
Observability dashboard
Simple React UI showing: consumer lag per group (how many messages behind is each consumer?), DLQ event count, payment throughput over time. Pulls data from the Redpanda Admin API. Consumer lag is Kafka's equivalent of your Sidekiq queue depth — the most important health signal.
3–4 days React + TypeScript Redpanda Admin API consumer lag metric
What you'll understand: consumer lag as a health signal (rising lag = consumer falling behind = problem), how Kafka exposes operational data, and why this dashboard is what separates "I ran Kafka" from "I ran Kafka in production."
Total POC time: ~3 weeks at a few hours/day. Phase 1–3 is the most important — that's where the concepts land. Phase 4–6 is where you build the interview story.
